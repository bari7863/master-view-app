# マスタデータ 引き継ぎ資料

この資料は、マスタデータアプリの詳細な開発経緯・仕様・注意点をまとめたものです。

GitHubで最初に見る概要は `README.md`、Claude Codeに守らせる作業ルールは `CLAUDE.md` に分けています。

---

## 1. アプリ概要

「マスタデータ」は、営業・マーケティングで使う企業情報を管理する社内向けWebアプリです。

主な目的は以下です。

- 大量の企業リストを一元管理する
- 企業情報の欠損を確認する
- 企業HPから不足情報を補完する
- 営業リストを精査する
- フォーム営業やテレアポに使いやすい状態に整える
- アポ率や営業効率の改善につなげる

関連ファイル。


## 5. クローリングの基本方針

このプロジェクトでは、クローリングの取得精度を最優先にします。

守るべき方針。

- 取得率を落とさない
- 正しい値を除外しない
- 会社と関係ない値を保存しない
- 1つの項目の改善で他項目を悪化させない
- 候補ページの探索範囲を狭めすぎない
- 会社概要ページだけに限定しすぎない
- Pythonでうまく取得できていたロジックは、できるだけTypeScript側にも反映する
- Playwrightの高速化は、精度が落ちない範囲で行う

## 7. 代表者名取得の考え方

代表者名は誤取得が起きやすい項目です。

見るべき文脈。

- 代表者
- 代表取締役
- 代表取締役社長
- 代表取締役会長
- 社長
- CEO
- COO
- 役員
- 役員紹介
- 会社概要
- 代表挨拶

重要な考え方。

- ラベルと氏名が同じ行にあるケースを見る
- ラベルの後ろにある氏名を取る
- 表形式、dl/dt/dd、テキスト行を読む
- 会社概要ページだけに限定しない
- 役職だけを保存しない
- 複数人がいる場合は、代表に近い氏名を優先する
- 除外処理で正しい氏名まで消さない

除外に注意する文言。

- プロフィール
- メッセージ
- インタビュー
- 企業ビジョン
- 経営理念
- ホーム
- ページの先頭へ戻る
- グループ
- 仕様
- 動画で知る
- 運営責任者
- 環境管理責任者

ただし、除外ワードを強くしすぎると正しい氏名も消える可能性があります。

## 12. クローリングjob状態管理

クローリングは長時間になるため、途中停止・再開・ブラウザ再読み込みへの対応が重要です。

主な状態。

```text
jobId
selectedFields
targets
nextIndex
total
processed
updated
skipped
failed
currentCompany
currentWebsiteUrl
running
paused
completed
startedAt
lastStartedAt
elapsedMs

```

注意点。

- 進捗をメモリだけに依存しない
- ブラウザ再読み込み後も状態を復元する
- 経過秒数や平均秒数が0に戻らないようにする
- 完了時は running=false にする前に経過時間を確定させる
- 「中止」を押した結果画面は再表示しない
- 「中止」を押していない場合は再表示できるようにする

## 13. 権限管理の考え方

権限管理は、ログインしたユーザーごとに使える機能を分けるための重要な機能です。

基本方針。

- 管理者は権限設定を変更できる
- 従業員は許可された機能だけ使える
- 画面で非表示にするだけでなく、API側でも守る
- ログイン中ユーザーの権限は `permissions/me` で確認する
- 権限設定の保存・取得は `permissions` APIで管理する
- 共通の権限定義や判定は `lib/master-data-permissions.ts` に集約する

注意点。

- 権限がないユーザーに、保存・削除・CSV抽出・クローリング開始などを実行させない
- UIだけで制限すると、APIを直接叩かれた場合に制御できない
- 権限キーを変更すると、保存済みの権限データが使えなくなる可能性がある
- 管理者自身の権限を誤って失うと、権限設定画面を操作できなくなる可能性がある
- Vercel環境でもローカル環境でも同じ権限判定になるようにする
- ログイン情報、権限情報、画面表示の3つがズレないようにする

権限管理を修正するときに確認するファイル。

```text
app/page.tsx
app/api/master_data/login/route.ts
app/api/master_data/permissions/route.ts
app/api/master_data/permissions/me/route.ts
lib/master-data-auth.ts
lib/master-data-permissions.ts
```

権限管理がDBに保存されている場合は、DBテーブルやSQLも確認してください。

## 14. worker構成

Vercelで長時間クローリングを行うと、関数実行時間やメモリの問題で不安定になります。

そのため、worker構成が導入されています。

基本方針。

- Vercelは画面と軽いAPIを担当
- 重いクローリングはworker側で実行
- workerが進捗や結果を保存する
- 画面側はAPIで進捗を取得する

ローカル環境では以下を優先します。

- workerを起動しなくてもクローリングできる
- `http://localhost:3000/` ではworker確認をスキップする
- 従来のローカルクローリング速度を維持する

Vercel公開URLでは以下を想定します。

- worker起動確認を行う
- worker未起動時は分かりやすいエラーを出す
- 長時間処理をVercel側に抱えさせすぎない

## 15. CSV取込・CSV抽出

### CSV取込

注意点。

- 大量CSVを一気に処理しすぎない
- フロント側で巨大データを持ちすぎない
- `Maximum call stack size exceeded` に注意する
- DBカラムとCSV列順を揃える

---

### CSV抽出

抽出パターンとして以下がありました。

- 全てのリスト
- 処理完了分
- 候補のみリスト
- 候補のみ

候補になった項目は、変更前と候補一覧を分けて出力する方針がありました。

CSV抽出はデータ持ち出しに関係するため、権限管理と連動している場合は、権限がないユーザーが抽出できないように注意してください。

## 16. DB変更時の注意点

DBカラムを追加・変更するときは、以下を必ず確認します。

- DBテーブル
- SQLファイル
- APIのSELECT
- APIのINSERT
- APIのUPDATE
- TypeScript型
- 画面表示
- フィルタ
- 並び替え
- CSV取込
- CSV出力
- 保存処理
- クローリング結果保存
- 権限管理に影響しないか

漏れがあると、表示されない、保存されない、CSVがズレる、TypeScriptエラーが出るなどの問題が起きます。

権限管理用のテーブルやカラムがある場合は、以下も確認してください。

- 権限の初期値
- 管理者の権限
- 従業員の権限
- 既存ユーザーへの影響
- Vercel環境のDBにも反映されているか

## 17. Vercel環境変数

Vercelには以下の環境変数を設定します。

```text
DATABASE_URL
MASTER_DATA_AUTH_TOKEN
MASTER_DATA_LOGIN_ADMINS
MASTER_DATA_LOGIN_EMPLOYEES
```

注意点。

- `.env.local` はVercelに自動反映されない
- Vercel側で別途登録が必要
- `DATABASE_URL` が既にある場合は新規追加ではなく編集する
- 各Keyには対応するValueを入れる
- `MASTER_DATA_AUTH_TOKEN` に `DATABASE_URL` の中身を入れない

権限管理が既存ログイン情報を使っている場合、追加の環境変数は不要な可能性があります。

ただし、コード側で権限管理用の新しい環境変数が追加されている場合は、Vercel側にも同じKeyを登録してください。

## 18. よくあったエラー

### `Maximum call stack size exceeded`

主な原因。

- 大量CSVを一気に処理
- 巨大配列の一括展開
- フロント側で重い処理
- 再帰的な処理

対応。

- chunk処理にする
- サーバー側処理に寄せる
- 一括setStateを避ける
- 大量データをブラウザメモリに持ちすぎない

---

### `Server is approaching the used memory threshold, restarting...`

主な原因。

- 長時間クローリング
- Playwrightの解放漏れ
- 大量HTMLの保持
- 大量データの一括処理

対応。

- workerへ分離する
- 1件ごとに不要データを破棄する
- browser/pageを確実にcloseする
- DBへ逐次保存する

---

### `名前 'xxx' が見つかりません`

主な原因。

- state定義漏れ
- import漏れ
- 関数の貼り付け位置ミス
- 変数スコープのズレ
- 不要コード削除による参照切れ

---

### `モジュール '@/lib/xxx' が見つかりません`

主な原因。

- importパスの誤り
- ファイル名の誤り
- 添付用ファイル名をそのままimportに使っている

例。

```text
lib_master-data-crawler.ts
```

のような添付用ファイル名を、そのままimportに使うとエラーになる可能性があります。

実際のファイル名は以下のような形式です。

```text
lib/master-data-crawler.ts
lib/master-data-auth.ts
lib/master-data-permissions.ts
```

---

### `プロパティ xxx は型に存在しません`

主な原因。

- 型定義に項目がない
- APIレスポンスに項目がない
- 初期値に項目がない
- DBカラムに項目がない

---

### 権限が反映されない

主な原因として考えられるもの。

- `permissions/me` のレスポンスが画面側に反映されていない
- 権限保存後に再取得していない
- ローカルとVercelで環境変数やDBが違う
- 管理者・従業員の判定がログイン処理と権限処理でズレている
- 画面側では非表示でもAPI側で制御されていない
- 権限キー名が画面側とAPI側で一致していない

対応。

- ログイン中ユーザーの権限取得結果を確認する
- `app/api/master_data/permissions/me/route.ts` を確認する
- `lib/master-data-permissions.ts` の権限定義を確認する
- 画面側の権限判定とAPI側の権限判定を確認する
- Vercel環境でも同じ設定・DBになっているか確認する

---

## 19. UI修正の方針

UI修正では以下を守ります。

- 既存デザインを大きく崩さない
- 横並びだったUIを勝手に縦並びにしない
- 小さい画面でも一覧や左メニューが見切れないようにする
- 長いラベルは文字サイズや幅で調整する
- ポップアップは中央付近に表示する
- 「お問い合わせフォームURL」など長い項目名は極力折り返さない
- メニュー折りたたみ時のロゴサイズは枠内でできるだけ大きくする
- 権限管理画面のボタンやチェック項目は見切れないようにする
- 権限設定の保存ボタンや全選択・選択解除ボタンの配置を崩さない

## 20. 不要ファイル・フォルダ削除の判断

削除前に必ず確認すること。

- importされていないか
- 画面からリンクされていないか
- APIから呼ばれていないか
- npm scriptsから参照されていないか
- Vercelやworkerで使われていないか
- 状態保存や復元に使われていないか
- 権限管理で使われていないか
- release配下のworker実行に影響しないか

```text
release/MasterCrawlWorker/

sql/add_column.sql
app/api/master_data/permissions/
lib/master-data-permissions.ts
```

ビルド済みファイル、配布用exe、SQLファイル、権限管理関連ファイルは、不要に見えても運用で使っている可能性があります。
